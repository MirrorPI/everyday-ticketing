## 이 프로젝트를 통해 경험하고 싶은 백엔드 실무 요소

이 프로젝트는 단순히 티켓팅 기능을 구현하는 것이 아니라, 신입 백엔드 개발자로서 취업 전 경험해봐야 할 기본기를 하나씩 쌓는 것을 목표로 한다.

## 1. API 설계

- REST API 원칙을 고려해 URL을 리소스 중심으로 설계한다.
- 좌석 선택은 좌석 수정이 아니라 선택 기록 생성으로 본다.
- HTTP Method의 의미를 고려해서 GET, POST, PATCH, DELETE를 구분한다.
- 예외 응답은 일관된 `code`, `message` 형식으로 반환한다.

## 2. 책임 분리와 코드 구조

- Controller는 요청과 응답만 담당한다.
- Service는 유스케이스 흐름과 트랜잭션을 담당한다.
- Repository는 DB 접근만 담당한다.
- Entity는 도메인 상태와 기본 규칙을 표현한다.
- DTO는 API 요청/응답 전용으로 사용한다.
- SOLID 원칙 중 특히 단일 책임 원칙을 의식하며 구현한다.

## 3. 테스트 코드

- 핵심 비즈니스 로직은 테스트 코드로 검증한다.
- 오픈 전 좌석 선택 실패를 테스트한다.
- 오픈 후 좌석 선택 성공을 테스트한다.
- 같은 사용자가 같은 이벤트에 두 번 참여할 수 없음을 테스트한다.
- 같은 좌석을 여러 사용자가 동시에 선택했을 때 한 명만 성공하는지 테스트한다.
- 좌석 선택 성공 후 랭킹에 반영되는지 테스트한다.

## 4. 트랜잭션과 동시성 제어

- 좌석 선택은 하나의 트랜잭션 안에서 처리한다.
- 좌석 선택 성공과 점수 반영이 따로 실패하지 않도록 한다.
- 동시에 같은 좌석을 선택하는 상황을 고려한다.
- 애플리케이션 검증뿐 아니라 DB Unique 제약조건으로도 중복을 방어한다.
- 필요하다면 낙관적 락, 비관적 락, Redis 방식의 차이를 비교해본다.

## 5. DB 설계와 제약조건

- ERD를 직접 설계한다.
- 사용자, 티켓팅 이벤트, 좌석, 좌석 선택 기록을 중심으로 테이블을 구성한다.
- 한 사용자는 한 이벤트에 한 번만 참여할 수 있도록 제약조건을 둔다.
- 한 좌석은 한 이벤트에서 한 명만 선택할 수 있도록 제약조건을 둔다.
- 나중에 히스토리 테이블을 추가해 선택 이력과 상태 변경 이력을 남기는 구조를 고민한다.

## 6. 로그

- 좌석 선택 요청의 성공과 실패를 로그로 남긴다.
- 실패 원인을 추적할 수 있도록 예외 로그를 남긴다.
- 단순히 에러가 났다는 로그가 아니라, 어떤 이벤트와 좌석에서 문제가 발생했는지 확인할 수 있게 한다.
- 개인정보나 민감한 정보는 로그에 남기지 않는다.

## 7. 성능 분석

- 단순히 기능이 동작하는 것에서 끝내지 않는다.
- 좌석 목록 조회, 랭킹 조회 쿼리를 분석한다.
- 필요하다면 `EXPLAIN` 또는 `EXPLAIN ANALYZE`로 실행 계획을 확인한다.
- 인덱스가 필요한 쿼리와 필요하지 않은 쿼리를 구분해본다.

## 8. 부하 테스트

- JMeter 또는 k6를 사용해 동시에 여러 사용자가 좌석을 선택하는 상황을 테스트한다.
- 같은 좌석에 요청이 몰렸을 때 한 명만 성공하는지 확인한다.
- 요청 수가 늘어났을 때 응답 시간이 어떻게 변하는지 기록한다.
- 부하 테스트 결과를 바탕으로 병목 지점을 정리한다.

## 9. 모니터링

- Spring Actuator를 사용해 애플리케이션 상태를 확인한다.
- 요청 수, 응답 시간, 에러 수 같은 기본 지표를 관찰한다.
- 가능하다면 Prometheus, Grafana를 붙여 간단한 대시보드를 구성한다.
- 운영 환경에서 문제가 발생했을 때 어디를 먼저 봐야 하는지 익힌다.

## 10. 인증과 보안

- 구글 로그인을 적용한다.
- 로그인한 사용자만 좌석을 선택할 수 있게 한다.
- 인증된 사용자 정보와 요청 사용자가 일치하는지 확인한다.
- API 요청에서 신뢰하면 안 되는 값과 서버에서 검증해야 하는 값을 구분한다.

## 11. 배포와 운영

- 로컬에서만 동작하는 프로젝트로 끝내지 않는다.
- Docker를 사용해 실행 환경을 정리한다.
- 가능하다면 EC2, RDS 등을 사용해 배포해본다.
- 운영 환경 변수와 로컬 설정을 분리한다.
- GitHub Actions를 사용해 간단한 CI/CD 흐름을 경험한다.

## 12. 문서화

- 기능을 만들 때마다 왜 그렇게 설계했는지 짧게 기록한다.
- 트레이드오프가 있었던 결정은 문서로 남긴다.
- 실패한 시도도 기록한다.
- 포트폴리오에는 단순 기능 목록보다 문제, 분석, 선택 이유, 결과를 중심으로 정리한다.
